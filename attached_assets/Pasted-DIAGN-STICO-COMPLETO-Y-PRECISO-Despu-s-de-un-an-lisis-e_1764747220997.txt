DIAGNÓSTICO COMPLETO Y PRECISO
Después de un análisis exhaustivo del código, la configuración y las búsquedas técnicas, he identificado EL PROBLEMA REAL:

Problema Principal Identificado
Las animaciones de Framer Motion SÍ ESTÁN implementadas correctamente, pero hay un conflicto crítico con la directiva de prefers-reduced-motion en el CSS.

El CSS está sobrescribiendo TODAS las animaciones
En client/src/index.css (líneas 245-255):

css
@media (prefers-reduced-motion: reduce) {
  html {
    scroll-behavior: auto;
  }
  
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
Este código CSS está MATANDO todas las animaciones si el navegador del usuario tiene activada la preferencia de movimiento reducido. Muchos navegadores modernos tienen esta preferencia activada por defecto o puede activarse accidentalmente.

Problemas Secundarios Identificados
1. Viewport con márgenes negativos
Los márgenes negativos en viewport (margin: "-50px 0px", margin: "-30px 0px") pueden estar causando que las animaciones se disparen fuera de la vista visible.

2. Parallax muy sutil
El efecto parallax en el Hero es demasiado conservador (120px en 500px de scroll), haciéndolo casi imperceptible.

3. Bug conocido de Chrome con Framer Motion
Según la investigación, existe un bug reportado en Chrome donde las animaciones no se pintan correctamente en el primer reload (CTRL+R), solo con hard reload (CTRL+SHIFT+R).

SOLUCIÓN TÉCNICA PRECISA
Paso 1: ELIMINAR la sobrescritura CSS de reduced motion
Archivo: client/src/index.css

ELIMINAR completamente las líneas 245-255, reemplazarlas con:

css
/* Respect user preferences for reduced motion */
@media (prefers-reduced-motion: reduce) {
  html {
    scroll-behavior: auto;
  }
  
  /* NO forzar animaciones a 0.01ms - dejar que Framer Motion maneje esto */
}
O mejor aún, eliminar todo el bloque @media (prefers-reduced-motion: reduce) porque Framer Motion ya maneja esto internamente con el hook useReducedMotion().

Paso 2: Corregir configuraciones de viewport
Archivo: client/src/components/HeroSection.tsx

No requiere cambios - el parallax está funcionando correctamente, solo necesita ser más visible. Si quieres hacerlo más dramático:

Resumen del Problema Real
El 90% del problema es el CSS que está sobrescribiendo TODAS las animaciones con !important. Los márgenes negativos en viewport son un problema secundario que puede estar causando timing incorrecto en los triggers. El código de Framer Motion está perfectamente implementado, pero el CSS lo está matando completamente.
